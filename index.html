Fluxo Básico de Trabalho

O ciclo que você vai repetir todo dia. Memorize essa sequência de 4 passos.

📁 Editar arquivo
→
git add
→
git commit
→
git push
⚡
Ciclo completo
Use toda vez que fizer alterações
# 1. Ver o que foi modificado git status # 2. Adicionar tudo ao stage git add . # 3. Registrar com mensagem git commit -m "sua mensagem aqui" # 4. Enviar para o GitHub git push origin main
💡 git add . adiciona todos os arquivos de uma vez. Para adicionar só um: git add nome-do-arquivo.html
🔍
Visualizando histórico
# Histórico completo git log # Resumido (mais usado) ⭐ git log --oneline # Com alterações detalhadas git log -p # Com gráfico de branches git log --graph
🔎
Comparando e inspecionando
# Ver detalhes do último commit git show # Ver commit específico git show 2ad48c0 # O que mudou (não adicionado) git diff # Comparar dois commits git diff abc..def
📖
Conceitos essenciais
TermoSignificado
HEADApelido para o commit mais recente da branch atual
Stage (index)Área intermediária — arquivos prontos para o commit
HashIdentificador único de cada commit (ex: 237cada)
diffFormato de diferenças: - = removido, + = adicionado

Branches — Linhas do Tempo Paralelas

Branches permitem desenvolver funcionalidades sem afetar o código principal.

main: A ── B ── C \ feature: D ── E ↑ git switch -c feature
🌿
Criando e navegando
# Listar todas as branches git branch # Criar nova branch e entrar nela ⭐ git switch -c nome-da-branch # Entrar em branch existente git switch nome-da-branch # Voltar para branch anterior (atalho) git switch - # Deletar branch (após merge) git branch -d nome-da-branch
🔀
git checkout — o "canivete suíço" dividido
A partir do Git 2.23, o git checkout foi dividido em dois comandos mais claros. Ele ainda funciona normalmente, mas os novos são preferidos.
ObjetivoComando antigoComando novo ⭐
Trocar de branchgit checkout featuregit switch feature
Criar e entrargit checkout -b novagit switch -c nova
Restaurar arquivogit checkout -- arquivogit restore arquivo
🧩
Fluxo completo: do início ao merge
# 1. Criar e entrar na branch de feature git switch -c feature/login # 2. Fazer alterações e commits git add . git commit -m "Adiciona tela de login" # 3. Voltar para a main git switch main # 4. Unir as mudanças git merge feature/login # 5. Deletar a branch (agora mergeada) git branch -d feature/login
🎮 Dica: Use o Visualizing Git para ver os branches se formando graficamente no navegador — sem instalar nada.

git restore
# Desfaz mudanças (1 arquivo) git restore arquivo.txt # Desfaz tudo no projeto (Ctrl+Z total) git restore . # Tira do stage (mantém mudanças) git restore --staged arquivo.txt # Viagem no tempo para commit X git restore --source=abc1234 arquivo
🗄️
git stash
# Guardar trabalho na "gaveta" git stash push -m "WIP: funcionalidade X" # Listar o que está guardado git stash list # Recuperar e remover da gaveta git stash pop # Aplicar sem remover git stash apply 1 # Limpar tudo git stash clear
🥞
Stash funciona como uma pilha
O último guardado é o primeiro a sair
↓ pop age aqui primeiro

Para recuperar um item específico: git stash pop 1 ou git stash apply 1

Tags & Releases — Versionando o Projeto

Marque pontos específicos do histórico como versões oficiais do seu projeto.

commits: A ── B ── C ── D ── E ↑ ↑ tag: v0.1.0 tag: v0.1.1 (save point!) (save point!) → A tag NUNCA se move. Aponta para o mesmo commit para sempre.
🔖
Tag simples
Só um apelido para o commit
# Cria na HEAD (commit atual) git tag v0.1.0 # Cria em commit específico git tag v0.1.0 502afb6 # Listar todas as tags git tag # Apagar tag localmente git tag -d v0.1.0
📝
Annotated Tag
Tag com mensagem, autor e data
# Cria annotated tag com mensagem git tag -a v0.1.1 -m "Versão 0.1.1" # O -a é opcional com -m git tag v0.1.1 -m "Versão 0.1.1" # Inspecionar annotated tag git tag -v v0.1.1 # Enviar todas as tags git push origin --tags
🚀
Tag simples vs Annotated
Tag SimplesAnnotated Tag
Aponta para commit✅✅
Mensagem própria❌✅
Guarda autor e data❌✅
Verificável com -v❌✅

Releases no GitHub
Publicação oficial de uma versão
Uma release é construída a partir de uma tag. Permite adicionar changelog, descrição da versão e arquivos para download (como um .exe compilado).

├── 📋 Release notes (changelog)
└── 📎 Assets
├── projeto.zip ← automático
├── projeto.tar.gz ← automático
└── app.exe ← você adiciona (opcional)

Cherry-pick & Blame

Selecione commits específicos entre branches e rastreie quem alterou cada linha.

🍒
git cherry-pick
Copiar um commit de outra branch
Quando usar: duas branches precisam da mesma alteração, mas você não quer fazer merge de tudo — só daquele commit específico.
branch2: A ── B(alteração) ── C branch1: A ── B' ← cópia do commit B aplicada aqui! O cherry-pick cria um NOVO commit — não move o original.
# 1. Ver commits de outra branch git log funcionalidade2 --oneline # 2. Estando na branch destino, aplicar o commit git cherry-pick 609bc79b23305a1db # 3. Enviar ambas as branches git push origin branch1 branch2
🔍
git blame
Quem alterou cada linha?
⚠️ Atenção ao mau uso: o nome "blame" significa "culpar". Use essa ferramenta para entender o código e saber com quem conversar — não para apontar culpados por bugs.
# Ver quem alterou cada linha git blame index.html # Inspecionar o commit encontrado git show 250c665
^250c665 (Rodrigo 2023-09-05 1) <!DOCTYPE html>
609bc79b (Vinicius 2023-12-23 14) <title>
↑ hash ↑ autor ↑ data ↑ linha ↑ conteúdo
💡 O ^ antes do hash indica o primeiro commit do projeto (commit original).
